# RAG 检索增强技术

[返回 Agent 开发笔记合集](./0、agent-dev-notes.md)

## 1、定义

RAG 是 Retrieval-Augmented Generation 的缩写，中文常叫“检索增强生成”。它的核心思路是：先从外部知识库检索相关材料，再把材料放入上下文，让模型基于材料生成回答。

RAG 不是让模型“记住”知识，而是让模型在回答时“带着资料作答”。

它解决的是模型知识有限、训练数据过期、私有资料不可见和答案缺少来源的问题。

## 2、为什么它重要

LLM 的训练知识并不等于当前事实，也不包含你的私有文档、代码库、产品手册和业务数据。

如果直接问模型，它可能凭印象回答。RAG 则会先把相关资料找出来，让模型基于资料回答。

在 Agent 系统里，RAG 可以作为一个工具，也可以作为上下文构建流程的一部分。它能降低幻觉风险，但前提是检索结果相关、来源可靠、上下文组织清楚。

## 3、基本流程

```text
文档收集 -> 切块 -> 向量化 / 索引 -> 检索 -> 重排 -> 放入上下文 -> 生成答案 -> 标注来源
```

常见组件如下：

| 组件 | 作用 |
| --- | --- |
| Document Loader | 把网页、PDF、Markdown、数据库内容读进来 |
| Chunking | 把长文档切成适合检索的小块 |
| Embedding | 把文本转成向量，方便相似度搜索 |
| Vector Store | 保存向量和元数据 |
| Retriever | 根据问题找相关片段 |
| Reranker | 对候选片段重新排序 |
| Generator | 让模型基于片段回答 |
| Citation | 标注答案依据来自哪里 |

RAG 的质量通常取决于检索，而不只是生成。

## 4、和长上下文的区别

| 方案 | 优点 | 风险 |
| --- | --- | --- |
| RAG | 成本可控，来源清晰，适合大规模知识库 | 检索错了就会答偏 |
| 长上下文 | 可以直接放入大量材料，减少切块损失 | 成本高，噪声多，重点可能被淹没 |
| 微调 | 适合学习风格、格式和稳定模式 | 不适合频繁更新的事实知识 |

RAG 和长上下文不是对立关系。很多系统会先检索，再把少量高相关材料放入长上下文。

## 5、一个 Agent 开发例子

假设要做一个“项目文档问答 Agent”，可以这样设计：

```text
1. 把 README、设计文档、ADR、API 文档切块入库
2. 用户提问时先检索相关片段
3. 按来源、时间、模块名做过滤
4. 把最相关片段放进上下文
5. 要求模型只基于材料回答
6. 每段答案附上来源链接
7. 找不到依据时明确说“不确定”
```

这样的 Agent 比单纯让模型凭记忆回答更可靠，也更容易让人检查。

## 6、常见误区

### 误区一：有 RAG 就不会幻觉

不会。RAG 只能提供材料，模型仍然可能误读、过度推断或引用错误片段。

### 误区二：向量检索越多越好

不是。召回太多会污染上下文。很多时候需要重排、过滤和摘要，而不是把所有片段都塞给模型。

### 误区三：切块随便切就行

切块会影响检索质量。代码、表格、文档标题、段落边界和元数据都要尽量保留语义结构。

### 误区四：RAG 只适合问答

RAG 也适合代码理解、客服、合规检查、报告生成、产品助手和企业搜索。

## 7、一句话总结

RAG 是让模型带着外部资料回答问题。它把“模型知道什么”变成“系统能检索到什么、能证明什么”。
