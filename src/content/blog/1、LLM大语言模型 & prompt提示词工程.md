# LLM 大语言模型与 Prompt 提示词工程

[返回 Agent 开发笔记合集](./0、agent-dev-notes.md)

## 1、定义

LLM 是 Large Language Model 的缩写，也就是大语言模型。它的核心能力是根据上下文理解、预测和生成语言，并在这个过程中表现出总结、翻译、推理、改写、代码生成等能力。

Prompt 是给模型看的任务输入。它可以是一句话，也可以是一份结构化任务说明，里面包含目标、背景、输入材料、约束条件、输出格式、示例和失败处理规则。

Prompt Engineering 是围绕提示词进行设计、调试和评估的方法。它不是“写一句神奇咒语”，而是在设计人、模型和系统之间的任务接口。

在 Agent 系统里，LLM 更像一个会理解、推理和表达的大脑。Prompt 则像给这个大脑的任务说明书，告诉它现在要做什么、依据什么做、不能做什么、结果要长什么样。

更简单地说：

```text
Prompt = 任务目标 + 必要上下文 + 行为边界 + 输出要求
```

## 2、为什么它重要

Agent 的很多行为都从模型输出开始。如果 Prompt 没有说清目标、边界和输出格式，模型就更容易猜测、跑偏，或者生成无法被程序处理的结果。

一个模糊的 Prompt 往往会把决策压力丢给模型。模型会自己补全缺失信息，但补全出来的内容不一定符合真实业务、用户意图或系统权限。

一个清晰的 Prompt 可以降低这种不确定性。它让模型知道优先级、判断依据、输出结构和遇到不确定情况时该怎么处理。

但 Prompt 也不是万能的。真实系统不能只靠一句“不要犯错”来保证安全，而要把 Prompt 和 Schema、工具权限、审批、日志、测试一起设计。

所以更准确的理解是：

> Prompt 负责表达意图，Harness 负责约束、验证和落地。

Prompt Engineering 是 Agent 开发的入口，但不是 Agent 工程的全部。

## 3、Prompt 的常见组成

一个 Prompt 不一定要包含所有部分，但可靠的 Prompt 通常会把关键信息分层写清楚。

| 组成 | 作用 |
| --- | --- |
| Role / 角色 | 告诉模型以什么身份完成任务 |
| Goal / 目标 | 说明这次任务最终要达成什么 |
| Input / 输入 | 给出需要处理的原始材料 |
| Context / 上下文 | 补充业务背景、历史信息和判断依据 |
| Constraints / 约束 | 说明不能做什么、必须遵守什么规则 |
| Process / 步骤 | 规定任务应该按什么顺序处理 |
| Output Format / 输出格式 | 要求输出 Markdown、JSON、表格、代码或结构化对象 |
| Examples / 示例 | 给出一两个输入输出样例，减少歧义 |
| Guardrails / 护栏 | 明确风险升级条件、失败处理方式和不确定时的行为 |

一个常见的 Agent prompt 不是只有用户问题，而是多层信息叠加后的结果：

```text
系统规则 + 开发者规则 + 用户请求 + 当前上下文 + 可用工具 + 输出格式
```

在应用里，这些信息通常来自不同位置。用户只看到自己输入的一句话，但模型实际收到的可能还包括系统指令、检索结果、工具定义、历史摘要和结构化输出要求。

## 4、一个提示词的基础结构

如果不知道怎么写，可以先使用下面这个结构：

```text
任务：你要完成什么？

背景：为什么要做？有哪些必要上下文？

输入：需要处理的材料是什么？

要求：必须遵守哪些规则、边界和偏好？

步骤：建议按什么顺序处理？

输出：结果要用什么格式？需要包含哪些字段？

失败处理：信息不足、冲突或无法完成时应该怎么说？
```

这不是唯一模板，但它覆盖了大多数日常任务。写 Prompt 时，最重要的不是句子华丽，而是让模型少猜。

比如一个“写文章摘要”的 Prompt 可以这样写：

```text
任务：请把下面这篇文章总结成一段中文摘要。

背景：摘要会展示在博客列表页，读者需要快速判断要不要点进去。

输入：<文章正文>

要求：
1. 不要添加文章中没有的信息。
2. 保留文章的核心观点和适合的关键词。
3. 语气自然，不要像广告文案。

输出：
一段 80 到 120 字的中文摘要。

失败处理：
如果正文信息不足，请说明“无法生成可靠摘要”，并指出缺少什么。
```

这个 Prompt 比“帮我总结一下”更稳定，因为它说明了使用场景、信息边界、风格要求和失败行为。

## 5、提示词应该怎么写

写 Prompt 可以按下面的顺序来。

### 第一步：先写清任务目标

先回答一个问题：你希望模型交付什么结果？

不要只写“帮我看看”“优化一下”“分析一下”。这些词太宽泛。更好的写法是说明交付物，比如“找出 3 个主要风险”“改写成适合发布的版本”“输出一个 JSON 分类结果”。

### 第二步：补足必要上下文

模型不会自动知道你的业务、读者、代码库、产品规则和前置讨论。只要这些信息会影响判断，就应该放进 Context。

上下文不等于越多越好。应该优先放和当前任务直接相关的信息，比如目标读者、业务规则、已有材料、限制条件和评价标准。

### 第三步：明确边界和约束

Prompt 里要写清楚什么可以做，什么不能做。

例如：“只基于给定材料回答”“不要编造引用”“不要修改无关文件”“只评论本次 diff”“如果不确定就标记为不确定”。

这些约束不能替代系统权限，但能减少模型在自然语言层面的误解。

### 第四步：指定输出格式

如果输出要被人读，可以指定 Markdown、表格、标题层级和语气。如果输出要被程序处理，就应该使用 JSON Schema、Zod 或其他结构化输出方式。

输出格式越明确，后续越容易复用、测试和自动化。

### 第五步：说明不确定时怎么办

很多坏结果不是因为模型不会，而是因为信息不足时仍然硬答。

可靠的 Prompt 应该告诉模型：缺少关键上下文时先说明缺口；证据不足时不要下结论；多个答案都可能时列出假设；任务不安全时请求确认或拒绝执行。

### 第六步：用示例减少歧义

示例是很强的约束。一个输入输出样例，往往比一大段抽象描述更能让模型理解你想要的结果。

示例不必很多。通常一到两个高质量示例就够了。示例越贴近真实任务，效果越稳定。

## 6、好 Prompt 和坏 Prompt 的区别

坏 Prompt 通常不是“短”，而是缺少关键决策信息。

| 坏写法 | 问题 |
| --- | --- |
| 帮我优化这段文字 | 不知道优化目标、读者、风格和改动幅度 |
| 分析这个项目 | 范围太大，不知道分析架构、风险、性能还是商业价值 |
| 写一个 JSON | 不知道字段、类型、必填项和失败处理 |
| 帮我修 bug | 不知道复现步骤、错误日志、期望行为和验证方式 |

更好的写法会把目标、输入、约束和输出放在一起。

```text
任务：请把下面这段产品介绍改写成更适合官网首屏的版本。

目标读者：中小企业的运营负责人。

要求：
1. 保留原文事实，不增加没有依据的承诺。
2. 语气专业、直接，不要夸张。
3. 中文输出，长度控制在 120 字以内。

输出：
只输出改写后的文案，不要解释修改过程。

原文：
<这里放原文>
```

这个版本并没有复杂很多，但模型已经知道该为谁写、怎么写、写多长、能不能添加新信息，以及最终输出什么。

## 7、一个 Agent 开发例子

比如要做一个“读 PR 并写 review”的 Agent，不要只写：

```text
帮我 review 这个 PR。
```

更可靠的 Prompt 会说明：

```text
目标：找出这个 PR 中的 bug、行为回归和缺失测试。

范围：只评论本次 diff，不评价无关历史代码。

上下文：PR diff、相关文件、项目规范、测试结果。

判断标准：
1. 优先报告会影响正确性、安全性、数据一致性的问题。
2. 风格问题只有在违反项目规范时才报告。
3. 不确定的问题放到“开放问题”，不要当成确定缺陷。

输出：
按严重程度列出 findings。
每个 finding 包含文件、行号、问题、原因和建议。

失败处理：
如果缺少关键上下文，先说明不能判断，不要猜。
```

这类 Prompt 更容易被 Agent Runtime 复用，也更容易和测试、review、日志系统配合。

在真正的 Agent 里，还可以继续把输出约束成结构化对象：

```json
{
  "findings": [
    {
      "severity": "high",
      "file": "src/example.ts",
      "line": 42,
      "problem": "这里描述问题",
      "recommendation": "这里描述建议"
    }
  ],
  "openQuestions": []
}
```

这样后续系统就可以把 review 结果渲染到页面、写入数据库，或者转成 GitHub 评论。

## 8、和相邻概念的区别

| 概念 | 重点 |
| --- | --- |
| Prompt | 这一次给模型看的输入 |
| Instructions | 更稳定的行为规则，通常来自系统或开发者 |
| Context | 模型本轮生成能看到的全部信息 |
| System Prompt | 优先级较高的系统层指令 |
| Developer Instructions | 应用开发者定义的流程、风格和工具规则 |
| User Prompt | 用户当前输入 |
| Prompt Template | 可复用的提示词模板 |
| Structured Output | 要求模型按固定结构返回结果 |
| Prompt Engineering | 设计、测试和改进 Prompt 的方法 |

Prompt 更像一次任务输入，Instructions 更像长期规则，Context 则是模型实际能看到的工作台。

在产品里，Prompt Template 很常见。它会把固定规则写好，再把用户输入、业务数据和检索材料填进变量。

```text
请根据以下资料回答用户问题。

规则：
- 只基于资料回答。
- 如果资料不足，说明无法判断。
- 回答使用中文。

资料：
{{retrieved_context}}

用户问题：
{{user_question}}
```

这类模板的价值是复用和稳定。它让同类任务使用同一套规则，而不是每次临时拼一句。

## 9、常见 Prompt 类型

不同任务适合不同写法。

| 类型 | 适合场景 | 关键点 |
| --- | --- | --- |
| 问答 Prompt | 基于资料回答问题 | 明确资料范围和不知道时的回答 |
| 总结 Prompt | 摘要、会议纪要、文章提炼 | 明确读者、长度、重点和不可添加信息 |
| 改写 Prompt | 文案、邮件、文章润色 | 明确风格、受众、保留信息和改动程度 |
| 分类 Prompt | 意图识别、工单分流 | 给出类别定义和置信度要求 |
| 抽取 Prompt | 从文本提取字段 | 使用结构化输出和缺失值规则 |
| 代码 Prompt | 写代码、修 bug、review | 给出环境、约束、测试和验收标准 |
| 工具调用 Prompt | Agent 选择工具和参数 | 明确何时调用、参数来源和失败处理 |

Prompt 的写法要服务于任务类型。让模型写文章和让模型生成可执行参数，应该使用不同的约束方式。

## 10、如何调试 Prompt

Prompt Engineering 不只是写 Prompt，还包括测试和迭代。

如果模型输出太泛，通常是目标和评价标准不够清楚。可以增加受众、使用场景、输出重点和反例。

如果模型经常编造，通常是资料边界和不确定处理不够清楚。可以要求只基于给定材料回答，并明确资料不足时的输出。

如果模型输出格式不稳定，通常是格式要求太松。可以使用结构化输出、Schema、示例和运行时校验。

如果模型忽略约束，可能是 Prompt 太长、优先级混乱，或者约束和用户请求冲突。可以把规则分层，把关键规则放到更高优先级的位置。

如果结果在不同输入上波动很大，就需要准备测试集。用一组代表性输入反复测试，比凭单次感觉改 Prompt 更可靠。

## 11、常见误区

### 误区一：Prompt 越长越好

不是。Prompt 应该足够完整，但也要有信息优先级。把所有材料都塞进去，会增加成本，也会让模型抓不住重点。

### 误区二：Prompt 能替代权限控制

不能。模型可能误解或忽略约束。删除文件、付款、发消息、写数据库这类动作必须靠系统权限、审批和沙盒控制。

### 误区三：Prompt Engineering 就是话术技巧

不是。好的 Prompt Engineering 要能被评估和复用，应该关注输入结构、输出结构、失败处理和验证方式。

### 误区四：模型越强，Prompt 越不重要

模型越强，能做的事越多，Prompt 反而更需要清晰地定义边界。否则系统会把模糊需求放大成不可控行为。

### 误区五：角色设定越复杂越好

不一定。“你是世界顶级专家”这类角色设定通常不如清晰的任务、上下文、约束和输出格式有用。

角色设定可以帮助确定语气和视角，但不能替代具体要求。

### 误区六：一个 Prompt 可以解决所有输入

复杂系统通常需要拆分任务。分类、检索、规划、执行、校验、总结，可能分别需要不同 Prompt 和不同输出结构。

把所有任务都塞进一个巨大 Prompt，会让系统更难测试，也更难定位错误。

## 12、一句话总结

LLM 是 Agent 的认知核心，Prompt 是给这个核心的任务接口。Prompt Engineering 的价值不是写漂亮话，而是把目标、上下文、边界、输出格式和失败处理设计清楚。

一个好的 Prompt 不一定很长，但应该让模型知道：

```text
我要做什么、依据什么做、不能做什么、结果长什么样、做不了时怎么办。
```
